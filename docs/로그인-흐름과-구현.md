# 로그인 흐름과 구현 정리

지금까지 만든 **로그인 기능**이 어떻게 동작하는지, 각 파일과 함수가 왜 그렇게 만들어졌는지 정리한 문서입니다.

---

## 1. 전체 흐름 (한눈에 보기)

```
[앱 실행]
    ↓
App: isLoggedIn 확인
    ↓
false → LoginPage 보여줌 (아이디/비번 입력)
true  → Index(메인) 보여줌

[사용자가 로그인 버튼 클릭]
    ↓
LoginPage: handleLogin 실행
    ↓
1) 입력값 비었는지 확인 → 비었으면 에러 메시지, 종료
2) getMockAuth(아이디, 비번) 호출 → 목 데이터와 비교
3) 일치하면 → authStore.login(아이디, 역할, 이름) 호출
4) 스토어에 isLoggedIn = true 저장
    ↓
App이 다시 그리면서 isLoggedIn이 true이므로 → Index(메인) 표시
    ↓
Index: 스토어에서 userName, name, role 읽어서 "○○ (역할)으로 로그인했습니다" 표시
```

**요약**: 로그인 여부는 **auth 스토어 한 곳**에서만 관리하고, App은 그 값에 따라 **LoginPage**와 **Index** 중 하나만 보여줍니다. 로그인 버튼은 “스토어의 login()을 호출”하는 역할만 합니다.

---

## 2. 왜 파일을 이렇게 나눴는지

| 담당 | 역할 |
|------|------|
| **auth.ts** | “로그인 상태”가 어떤 데이터를 가지는지 **타입만** 정의 (실제 값 없음) |
| **authStore.ts** | 로그인 상태 **값**과 **login/logout 함수**를 가지고, 앱 전체에서 공유 |
| **auth.mock.ts** | **목(가짜) 계정 목록**과 “아이디/비번이 맞는지 검사”하는 함수 (실제 API 대신) |
| **App.tsx** | “로그인됐으면 메인, 안 됐으면 로그인 페이지” **화면만** 결정 |
| **LoginPage** | 입력창, 버튼, **버튼 눌렀을 때 검사 후 스토어에 로그인 처리** |
| **Index** | 메인 페이지. 스토어에서 **누가 로그인했는지** 읽어서 표시 |

나중에 실제 서버 API를 붙이면, **auth.mock.ts 대신** “API 호출 후 결과로 login() 호출”하는 부분만 바꾸면 됩니다.

---

## 3. 파일별 상세 설명

### 3-1. auth.ts — “로그인 상태”의 모양 정의

**역할**: 로그인했을 때 **어떤 정보를 저장할지**를 타입으로만 정해 둔 파일입니다. 값은 없고 “형태”만 정의합니다.

```ts
// AuthState = 로그인 상태가 가질 수 있는 필드들
userName: string | null   // 로그인할 때 입력한 아이디
name?: string | null      // 표시용 이름 (목 데이터에서 옴)
role?: UserRole | null    // admin / manager / member
isLoggedIn: boolean       // 로그인 여부 (true면 메인, false면 로그인 페이지)
```

**왜 이렇게 했는지**: 스토어에 “무슨 키를 넣을 수 있는지”를 한곳에서 관리하면, 나중에 필드를 추가/수정할 때 타입 오류로 실수를 바로 잡을 수 있습니다.

---

### 3-2. authStore.ts — 실제 “로그인 상태” + login / logout

**역할**:  
- 앱 전체에서 **하나만** 있는 “로그인 상태”를 Zustand로 보관하고,  
- **login**, **logout**으로 그 상태를 바꾸는 곳입니다.

**저장되는 값 (초기)**  
- `userName: null`, `name: null`, `role: null`, `isLoggedIn: false`  
→ “아무도 로그인 안 한 상태”

**login(userName, role, name?)**  
- 인자: 로그인한 **아이디**, **역할**, (선택) **표시 이름**  
- 하는 일: 스토어를 `{ userName, name, role, isLoggedIn: true }` 로 바꿉니다.  
- **왜 이렇게 했는지**: 로그인 “성공”이 뭔지는 LoginPage나 mock이 결정하고, 스토어는 “성공했다고 알려주면 상태만 갱신”하는 역할만 하도록 했습니다. 그래서 **비밀번호는 받지 않고**, “누가, 어떤 역할로 로그인했는지”만 받습니다.

**logout()**  
- 하는 일: 다시 `userName: null`, `name: null`, `role: null`, `isLoggedIn: false` 로 되돌립니다.

**사용하는 쪽**  
- `useAuthStore((state) => state.isLoggedIn)` → 로그인 여부만 필요할 때 (App)  
- `useAuthStore((state) => state.login)` → 로그인 처리할 때 (LoginPage)  
- `useAuthStore((state) => state.userName)` 등 → 메인에서 “누가 로그인했는지” 보여줄 때 (Index)

---

### 3-3. auth.mock.ts — 목 로그인용 계정 + 검사 함수

**역할**:  
- 실제 서버 대신 **고정된 계정 목록**으로 “이 아이디/비번이 맞는지” 판단하고,  
- 맞으면 **역할(role)** 과 **표시 이름(name)** 을 돌려줍니다.

**MOCK_ACCOUNTS**  
- 키: 로그인 **아이디** (소문자로 비교할 예정)  
- 값: `{ password, role, name }`  
  - admin → 비번 1234, 역할 admin, 이름 "최고관리자"  
  - manager → 비번 1234, 역할 manager, 이름 "팀장"  
  - member → 비번 1234, 역할 member, 이름 "일반유저"

**getMockAuth(username, password)**  
1. `username`을 **공백 제거 + 소문자**로 정규화합니다. (대소문자 구분 안 하기 위함)  
2. `MOCK_ACCOUNTS[trimmed]` 로 해당 아이디의 계정 정보를 가져옵니다.  
3. 계정이 없거나, **비밀번호가 일치하지 않으면** → `null` 반환 (로그인 실패).  
4. 일치하면 → `{ role: account.role, name: account.name }` 반환 (로그인 성공).

**왜 이렇게 했는지**:  
- “누가 로그인할 수 있는지”, “비번이 뭔지”는 **이 파일 한 곳**에서만 관리하면 됩니다.  
- 나중에 API를 붙일 때는 `getMockAuth` 대신 “API 호출 후 응답으로 role, name 받아서 사용”하는 함수로만 바꾸면 됩니다.

---

### 3-4. App.tsx — 로그인 여부에 따라 화면만 전환

**역할**:  
- **isLoggedIn**이 true면 **Index(메인)**, false면 **LoginPage**만 보여줍니다.  
- 라우터 없이 “한 화면만 바꾸는” 구조입니다.

```tsx
const isLoggedIn = useAuthStore((state) => state.isLoggedIn);
return <>{isLoggedIn ? <Index /> : <LoginPage />}</>;
```

**동작**:  
- 처음엔 `isLoggedIn === false` → LoginPage만 보임.  
- 어딘가에서 `login(...)`이 호출되면 스토어가 바뀌고, React가 App을 다시 그립니다.  
- 그때 `isLoggedIn === true` 이므로 `<Index />`가 보입니다.  
- **별도로 “페이지 이동” 코드를 넣을 필요 없이**, 스토어 값만 바꾸면 화면이 바뀝니다.

---

### 3-5. LoginPage — 입력 + 버튼 + “검사 후 로그인”

**역할**:  
- 아이디/비밀번호 **입력**  
- 로그인 **버튼**  
- 버튼을 눌렀을 때 **입력값 검사 → 목 데이터 검사 → 스토어에 로그인 반영**

**state**  
- `username`, `password`: input에 입력한 값 (controlled component)  
- `error`: 에러 메시지 (“비어 있음”, “아이디/비번 틀림”)

**handleLogin (흐름)**  
1. `setError("")` 로 이전 에러를 지웁니다.  
2. **아이디나 비번이 비어 있으면**  
   - `setError("아이디와 비밀번호를 입력해주세요.")`  
   - `return` → login 호출 안 함  
3. **getMockAuth(username, password)** 호출  
   - `null`이면: 목 데이터에 없는 조합 → `setError("아이디 또는 비밀번호를 확인해주세요.")` 후 `return`  
   - `{ role, name }` 이면: 성공  
4. **login(username.trim(), auth.role, auth.name)** 호출  
   - authStore의 `login`이 실행되고, `isLoggedIn`이 true가 됨  
   - App이 다시 그리면서 Index(메인)이 보이게 됨

**input과 state 연결**  
- `value={username}`, `onChange={(e) => setUsername(e.target.value)}`  
  → 입력할 때마다 `username` state가 바뀌고, 그 값이 다시 input에 반영됩니다 (controlled component).  
- 비밀번호도 같은 방식으로 `password` state와 연결됩니다.

**정리**:  
- “검사”는 **getMockAuth**가 하고,  
- “실제로 로그인 처리”는 **authStore.login**이 합니다.  
- LoginPage는 “입력값 모으기 → 검사 요청 → 성공 시 login 호출”만 담당합니다.

---

### 3-6. Index — 메인에서 “누가 로그인했는지” 표시

**역할**:  
- 스토어에서 **userName**, **name**, **role**을 읽어서  
- “**(이름 또는 아이디)** (역할)으로 로그인했습니다.” 문구를 보여줍니다.

```tsx
const displayName = name ?? userName;
// name이 있으면 name, 없으면 userName 사용
```

**왜 name ?? userName 인지**:  
- 목 데이터에서 **name**(예: "최고관리자")을 넣어 두었기 때문에, 있으면 그걸 보여주고,  
- 없을 때를 대비해 **userName**(로그인 아이디)을 대신 쓰도록 했습니다.

---

## 4. 로그인 버튼 한 번 눌 때의 데이터 흐름 (요약)

1. 사용자가 아이디/비번 입력 후 **로그인 버튼 클릭**  
2. **handleLogin** 실행  
   - `username`, `password` state 사용  
   - `getMockAuth(username, password)` → `null` 또는 `{ role, name }`  
3. 성공 시 **login(username, role, name)** 호출  
   - authStore 안의 값이 바뀜 (`isLoggedIn: true` 등)  
4. **useAuthStore**를 쓰는 컴포넌트(App, Index)가 **다시 렌더**  
5. App은 `isLoggedIn === true` 이므로 **Index** 렌더  
6. Index는 스토어에서 **userName, name, role**을 읽어서 화면에 표시  

이렇게 **스토어 한 곳**을 바꾸는 것만으로, App과 Index가 자동으로 “로그인된 상태”를 반영하게 됩니다.

---

## 5. 나중에 확장할 때

- **실제 로그인 API**를 쓰려면:  
  - `getMockAuth` 대신 “API 호출 → 응답에서 role, name 등 파싱 → login(…) 호출”하는 함수를 만들고,  
  - LoginPage의 handleLogin에서 그 함수만 호출하도록 바꾸면 됩니다.  
- **라우터**를 쓰려면:  
  - App에서 `<Index />` / `<LoginPage />` 대신 `<Routes>` + `<Route>` 로 경로를 나누고,  
  - 필요하면 “로그인 안 됐을 때 /login으로 보내기” 같은 가드를 추가하면 됩니다.

이 문서는 위 흐름과 결정을 기준으로 작성되었습니다. 코드를 바꾸면 이 문서도 같이 업데이트하면 이해하기 좋습니다.
