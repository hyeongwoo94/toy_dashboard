# 로그인 기능 구현 시 갖어야 할 사고방식

누군가 "로그인 기능 만들어 줘"라고 했을 때, **어떤 순서로 생각하면 되는지** 정리한 문서입니다.  
기능을 단계별로 쪼개서, 빠짐없이 구현할 수 있도록 합니다.

---

## 사고 순서 요약

1. **로그인 상태에 따른 페이지 분기 (삼항 연산자)**
2. **버튼에 상태 변화 함수 연결**
3. **목데이터 생성 (또는 API 연결 준비)**
4. **목데이터와 입력값 비교**
5. **비교 결과에 따른 로그인 상태 변화**
6. **잘못 입력했을 때 에러 메시지**

아래는 각 단계를 “무엇을 생각하고, 왜, 어떻게 할지” 순서로 풀어 쓴 것입니다.

---

## 1. 로그인 상태에 따른 페이지 분기 (삼항 연산자)

**생각할 것**  
“로그인했는지 안 했는지”를 **한 곳(스토어)**에서 관리하고, 그 값에 따라 **보여줄 화면**을 나눠야 한다.

**왜**  
- 로그인 전에는 로그인 페이지만, 로그인 후에는 메인만 보여주려면 “지금 로그인됐는지”라는 **단일 기준**이 필요하다.  
- 그 기준을 스토어에 두고, App에서 **삼항 연산자**로 “true면 메인, false면 로그인 페이지”만 결정하면 된다.

**이 프로젝트에서**  
- `authStore`에 `isLoggedIn`을 두고,  
- `App.tsx`에서 `useAuthStore((state) => state.isLoggedIn)`으로 읽어서  
  `isLoggedIn ? <Index /> : <LoginPage />` 로 분기한다.

**정리**  
먼저 “로그인 상태 한 개”와 “그에 따른 화면 하나만 바꾸는 구조”를 잡는다.

---

## 2. 버튼에 상태 변화 함수 연결

**생각할 것**  
로그인 **버튼**을 눌렀을 때, “실제로 로그인된 상태로 바꿔 주는 함수”가 **한 번** 호출되어야 한다.  
그 함수는 스토어에 있어야 하고, 버튼은 그 함수만 부르면 된다.

**왜**  
- 상태는 스토어가 갖고, 화면은 그 상태를 구독해서 그리므로,  
  “로그인 처리”도 **스토어의 함수(예: login)**가 담당하는 게 맞다.  
- 버튼의 역할은 “그 함수를 호출하는 것”까지다.

**이 프로젝트에서**  
- `useAuthStore((state) => state.login)`으로 **login 함수만** 가져와서  
- 로그인 버튼(또는 form submit)에서 `login(name, role)`을 호출한다.  
- 나중에 “언제 login을 호출할지(검사 통과 시에만)”는 4·5단계에서 정한다.

**정리**  
버튼/폼에는 “상태를 바꾸는 함수(login)”를 연결해 두고, “언제 호출할지”는 다음 단계에서 정한다.

---

## 3. 목데이터 생성 (또는 API 연결)

**생각할 것**  
“이 아이디·비밀번호가 맞는지” 판단하려면 **비교할 대상**이 필요하다.  
처음에는 **목데이터(배열/객체)**를 만들고, 나중에 API로 바꿀 수 있게 한 군데에만 두는 게 좋다.

**왜**  
- 서버 없이도 로그인 플로우를 만들 수 있고,  
- 나중에 API를 붙일 때는 “비교하는 부분”만 바꾸면 된다.

**이 프로젝트에서**  
- `features/user/mockUsers.ts`에 `User` 타입에 맞는 목데이터 배열을 둔다.  
- 각 항목에는 `loginId`, `password`, `name`, `role` 등이 들어 있다.  
- 실제 API로 바꿀 때는 이 파일을 “API 호출 결과”로 대체하거나, 비교 로직만 API 응답을 쓰도록 바꾸면 된다.

**정리**  
“검사할 계정 목록”을 한 곳에 두고, 나중에 API로 교체할 수 있게 한다.

---

## 4. 목데이터와 입력값 비교

**생각할 것**  
사용자가 입력한 **아이디·비밀번호**와 목데이터(또는 API 결과)를 **비교**해서,  
“이 조합이 맞는지” 판단하는 로직이 필요하다.  
이 비교는 **login을 호출하기 전**에 이루어져야 한다.

**왜**  
- 맞을 때만 스토어의 `login`을 호출해야 하므로,  
  “비교 → 통과 시에만 login 호출” 순서가 되어야 한다.

**이 프로젝트에서**  
- 로그인 폼 제출 시:  
  - `mockUsers`에서 `loginId`로 먼저 유저를 찾고,  
  - 없으면 “아이디 오류”,  
  - 있으면 그 유저의 `password`와 입력 비밀번호를 비교해서  
    다르면 “비밀번호 오류”, 같으면 다음 단계(로그인 상태 변화)로 넘어간다.  
- 필요하면 `checkLogin(loginId, password)` 같은 함수로 “아이디+비번 일치 여부”만 묶어 둘 수도 있다.

**정리**  
입력값과 목데이터(또는 API)를 비교하고, “맞다/틀리다”를 구분할 수 있게 한다.

---

## 5. 비교 후 로그인 상태 변화

**생각할 것**  
비교 결과 **“맞다”**일 때만, 스토어의 **login**을 호출해서 `isLoggedIn`과 사용자 정보(name, role 등)를 바꾼다.  
이렇게 하면 1단계에서 만든 “로그인 상태에 따른 페이지 분기”가 자동으로 동작한다.

**왜**  
- 로그인 상태는 스토어 한 곳에서만 관리하기로 했으므로,  
  “실제로 로그인 처리”는 **login 호출 한 번**으로 끝나야 한다.  
- login이 호출되면 App이 다시 그려지고, 삼항 연산자 때문에 메인 페이지가 보이게 된다.

**이 프로젝트에서**  
- 비교해서 일치하는 유저가 있으면 `login(user.name, user.role)` 호출.  
- auth 스토어에 `name`, `role`, `isLoggedIn: true` 등이 저장되고,  
  App에서 `isLoggedIn ? <Index /> : <LoginPage />` 때문에 메인이 보인다.

**정리**  
“비교 통과 → login 호출”만 하면, 나머지는 스토어와 삼항 연산자가 처리한다.

---

## 6. 잘못 입력했을 때 에러 메시지

**생각할 것**  
- **비어 있을 때**: 아이디/비밀번호가 비어 있으면 제출하지 못하게 하고, 해당 필드에 에러 메시지를 보여준다.  
- **목데이터와 비교해서 틀릴 때**:  
  - 아이디가 없으면 → 아이디 필드에만 에러  
  - 아이디는 있는데 비밀번호가 다르면 → 비밀번호 필드에만 에러  
  (원하면 “아이디 또는 비밀번호를 확인해 주세요”처럼 하나로 통일할 수도 있다.)

**왜**  
- 사용자에게 “왜 로그인이 안 됐는지” 알려주면 UX가 좋아지고,  
  어느 필드가 문제인지 알려주면 수정하기 쉽다.

**이 프로젝트에서**  
- 제출 시(handleSubmit)  
  - 빈 값 검사 → 해당 필드만 `setLoginIdError(true)` 또는 `setPasswordError(true)`  
  - 목데이터 비교 후  
    - 아이디 없음 → `setLoginIdError(true)`  
    - 비밀번호 불일치 → `setPasswordError(true)`  
- input 컴포넌트는 `showError`가 true일 때 에러 문구에 `on` 클래스를 붙여서 메시지를 보이게 한다.

**정리**  
“비어 있음”과 “목데이터와 불일치”를 구분하고, 필드별로 에러 상태를 켜서 메시지를 보여준다.

## 전체 순서 한 번에 (체크리스트)

| 순서 | 할 일 | 확인 |
|------|--------|------|
| 1 | 로그인 상태 한 곳(스토어)에 두고, App에서 삼항으로 로그인 페이지 / 메인 분기 | |
| 2 | 로그인 버튼(또는 form submit)에 스토어의 login 호출 연결 | |
| 3 | 목데이터(또는 API)로 “검사할 계정 목록” 준비 | |
| 4 | 제출 시 입력값과 목데이터(또는 API) 비교 | |
| 5 | 비교 통과 시에만 login 호출 → 로그인 상태 변화 | |
| 6 | 빈 값 / 아이디 오류 / 비밀번호 오류일 때 에러 메시지 표시 | |

이 순서대로 생각하면 “로그인 기능 만들어 줘”라고 했을 때 빠짐없이 구현할 수 있다.

---

## 이 프로젝트에서의 파일 역할 (참고)

| 담당 | 역할 |
|------|------|
| **auth.ts** | 로그인 상태의 **타입** 정의 (name, role, isLoggedIn 등) |
| **authStore.ts** | 로그인 상태 **값**과 **login / logout** 함수, 앱 전역 공유 |
| **mockUsers.ts** | 목데이터(유저 목록). 나중에 API로 대체 가능 |
| **checkLogin.ts** (선택) | 아이디+비밀번호로 유저 찾기/비교 로직 묶어 둔 함수 |
| **App.tsx** | `isLoggedIn`으로 로그인 페이지 / 메인 **삼항 분기** |
| **LoginPage** | 입력, form submit, 유효성·비교 후 login 호출, 에러 state로 메시지 표시 |
| **Index** | 스토어에서 name, role 읽어서 “누가 로그인했는지” 표시 |

이 문서는 “로그인 기능 구현 시 사고방식”을 기준으로 정리한 것입니다. 코드가 바뀌면 이 문서도 같이 업데이트하면 이해하기 좋습니다.
