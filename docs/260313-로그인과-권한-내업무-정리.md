## 260313 작업 정리 — 로그인, 권한, 내 업무 필터, 상태 관리

### 1. 로그인/유저 구조 리팩터링

- **User 타입 정리**
  - `id` 제거, `username` → `loginId` 로 변경, `name`, `role` 유지.
- **auth 상태 재설계**
  - `AuthState`를 `name`, `role`, `isLoggedIn` 중심으로 단순화.
  - 로그인 후 화면에서 실제로 쓰는 값은 **이름(name)과 역할(role)** 이라는 점을 명확히 함.
- **로그인 흐름**
  - `LoginPage`에서 `loginId`, `password`를 `useState`로 관리.
  - `mockUsers` + `checkLogin(loginId, password)`로 아이디/비밀번호 비교:
    - 아이디 없음 → 아이디 필드만 에러
    - 아이디는 있고 비밀번호만 틀림 → 비밀번호 필드만 에러
  - 비교 통과 시 `login(user.name, user.role)` 호출 → authStore에 로그인 상태 반영.
- **폼/UX 정리**
  - `form onSubmit` + 버튼 `type="submit"` 으로 엔터 키로도 로그인 가능.
  - `showError` 플래그로 에러 메시지에만 `on` 클래스를 토글해 스타일 제어.

### 2. 내 업무 필터링과 탭/대시보드 구조

- **TaskList 탭 구조**
  - 상단에 **전체 / 내 업무** 탭 버튼.
  - `useLocation` + `URLSearchParams`로 `/task`, `/task?tab=mine` 쿼리에서 탭 상태 계산:
    - `tab === "all"` → 전체
    - `tab === "mine"` → 내 업무
  - `visibleTasks` 계산:
    - 전체 탭 또는 아직 로그인 이름이 없으면 → `tasks`
    - 내 업무 탭 + 로그인 이름이 있으면 → `tasks.filter(t => t.assigneeId === name)`
- **메인 대시보드와 연동**
  - `MainList`:
    - `useTasks()`로 전체 업무를 가져와 상위 N개만 리스트에 표시.
    - 더보기 버튼 → `/task` 로 이동 (전체 탭).
  - `MyWorkList`:
    - `useTasks()` + `assigneeId === 로그인 name` 으로 **내 업무만 필터**.
    - 더보기 버튼 → `/task?tab=mine` 으로 이동 (내 업무 탭이 켜진 상태).

### 3. TaskView 권한과 상태(status) 변경 로직

- **로그인 정보 활용**
  - `useAuthStore(state => state.name)`, `useAuthStore(state => state.role)` 처럼 각각 구독해서 React의 `getSnapshot` 경고/무한 루프를 피함.
- **상태 변경 권한 (canChangeStatus)**
  - 아래 둘 중 하나일 때만 상태 버튼을 클릭 가능하게 설계:
    1. `role === "admin"` (관리자)
    2. `task.assigneeId === 로그인한 name` (해당 업무 담당자 본인)
  - `TaskState`에 `readOnly={!canChangeStatus}` 적용 + `onChange` 내부에서도 `if (!canChangeStatus) return`으로 한 번 더 방어.
- **수정/삭제 버튼 권한 (canEditOrDelete)**
  - 아래 조건 중 하나면 상단의 **수정하기 / 삭제하기** 버튼을 보여줌:
    1. `role === "admin"` (관리자)
    2. 로그인한 이름이 작성자(`task.authorId === name`)
    3. 로그인한 이름이 담당자(`task.assigneeId === name`)
  - 목록 버튼은 항상 보이도록 분리해서, **view는 누구나 볼 수 있지만 수정/삭제는 제한**하는 구조로 만듦.

### 4. API에 없는 status 필드를 로컬에서만 관리하기

- **문제 인식**
  - DummyJSON API에는 `status`, `작성자 이름`, `담당자 이름`, `작성일`, `마감일` 같은 필드가 없음.
  - 기존에는 `updateTask`가 DummyJSON에도 PUT 요청을 보내고, 응답을 다시 `Task`로 매핑한 뒤 `upsertLocalTask`로 localStorage에 저장.
  - 하지만 DummyJSON은 데모 API라 상태가 영구 반영되지 않아, 새로고침 시 상태가 다시 초기값처럼 보일 수 있음.
- **TaskStore 설계 확인**
  - `loadTaskStore` / `saveTaskStore` / `upsertLocalTask` 로 localStorage에 TaskStore를 관리.
  - `getTasks`, `getTaskById`에서:
    - 서버 응답을 `mapDummyTodoToTask`로 변환한 뒤
    - localStorage의 수정/생성/삭제 결과와 병합해서 화면에 사용할 최종 Task 리스트를 만듦.
- **status 전용 로컬 업데이트 함수 추가**
  - `features/task/api.ts` 에 다음 헬퍼 추가:

    ```ts
    export function updateTaskStatusLocal(id: string, status: TaskStatus): void {
      const store = loadTaskStore();
      const existing = store.byId[id];
      if (!existing) return;
      const next: Task = { ...existing, status };
      upsertLocalTask(next);
    }
    ```

  - 이 함수는 **네트워크 요청 없이** localStorage(TaskStore)에만 status를 업데이트함.
- **TaskView에서의 사용**
  - 상태 버튼 클릭 시:

    ```ts
    if (!id || !canChangeStatus) return;
    setStatus(value); // 화면 즉시 반영
    updateTaskStatusLocal(id, value as TaskStatus); // localStorage에만 저장
    ```

  - 이렇게 하면:
    - 화면에서는 즉시 바뀌고,
    - 새로고침 후에도 `getTaskById`/`getTasks`가 localStorage의 status를 가져오기 때문에 **상태가 유지**됨.
  - 중요한 포인트: **API 스키마에 없는 필드는 API 응답에 억지로 맞추지 말고, 로컬 스토어에서만 관리한다**는 패턴.

### 5. TaskEdit에서 로그인 유저를 작성자로 자동 입력

- **신규 등록 모드(`/task/edit`)**
  - `useAuthStore(state => state.name)` 으로 현재 로그인한 이름을 `currentUserName`으로 가져옴.
  - `useEffect`에서 id가 없을 때:

    ```ts
    setCreatedDay(today());          // 작성일 = 오늘
    if (currentUserName) setAuthorId(currentUserName); // 작성자 = 로그인 이름
    setIsLoading(false);
    ```

  - 이렇게 해서 “업무 등록하기” 화면을 열면 **작성자에 현재 로그인 계정 이름이 기본값**으로 들어가도록 함.
- **수정 모드(`/task/edit/:id`)**
  - 기존 Task의 `authorId`를 그대로 사용해, 등록 당시 작성자가 누구였는지를 유지.

### 6. CommonList에서 상태 컬럼 스타일링 준비

- **목표**
  - 공통 리스트에서 상태(status)에 따라 다른 색/스타일을 입힐 수 있게 준비.
- **구현**
  - `CommonList` 내부에서:
    - `col.label === "상태"` 인 컬럼을 찾아,
    - 해당 셀 텍스트(예: "요청", "진행", "검토", "완료")를 기준으로 클래스명을 붙임.

    ```ts
    const getStatusClass = (text: React.ReactNode) => {
      if (typeof text !== "string") return "";
      if (text === "완료") return "status-done";
      if (text === "진행" || text === "진행중") return "status-progress";
      if (text === "요청") return "status-request";
      if (text === "검토") return "status-review";
      return "";
    };
    ```

    ```tsx
    <div
      className={`_item_td ${
        col.label === "상태" ? getStatusClass(row.cells?.[ci]) : ""
      }`}
    >
      {row.cells?.[ci]}
    </div>
    ```

  - SCSS에서 `.status-done`, `.status-progress` 등을 이용해 색상/뱃지 스타일을 입힐 수 있는 구조를 만들어 둠.

### 7. 오늘 정리하면서 느낀 포인트 (블로그용 메모)

- **로그인/권한/필터/상태**는 모두 “전역 상태(auth, task) + 화면에서의 해석” 구조로 풀었다.
- API 스키마에 없는 정보를 클라이언트에서 관리할 때는:
  - “API와 동일한 Task 타입”을 억지로 맞추려 하기보다,
  - **로컬 TaskStore를 하나 더 두고 merge하는 쪽이 훨씬 안정적**이라는 점.
- 권한 로직은:
  - 비즈니스 룰을 `canChangeStatus`, `canEditOrDelete` 같은 **이름 있는 boolean**으로 먼저 표현하고,
  - JSX에서는 그 boolean만 사용하게 하면 읽기 쉽고, 테스트/변경도 수월하다는 점.
